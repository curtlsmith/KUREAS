Exercise 1: Creating a System and Adding Components

In this exercise you will create a new system model, define its components (SSCs), and assign failure modes with reliability data.

1

Open KUREAS-SYS.html in your browser. You will see a 6-pane layout: Pane A (Component List) on the far left, Pane B (System Info bar) across the top, Pane C (SSCs in System) and Pane D (Supercomponents) on the mid-left, Pane E (“How the System Works” logic editor) in the center, and Pane F (Edit SSC) on the right.

2

Name your system. In Pane B (the blue bar), enter System Name = Emergency Cooling System, Acronym = ECS. Click the Description field and enter a brief description of the system’s function.

3

Add your first SSC. In Pane C, click the +SSC button. In the dialog, enter: Name = PUMP-A, Description = Train A emergency cooling pump, SSC Type = Active, Component Type = Motor-Driven Pump. For the failure mode, enter: Failure Mode = Fails to Start, then set Failure Model = FM0: Direct Probability, Distribution = Lognormal, Mean = 3E-3, Error Factor = 5. Click Save.

4

Add a second failure mode. Select PUMP-A in Pane C, then click +FM. Enter: Failure Mode = Fails to Run, Failure Model = FM2: Fails While Operating, enter Lambda (failure rate) = 1E-5 per hour and Mission Time = 24 hours. The probability is computed automatically. Click Save.

5

Add remaining components. Repeat to add PUMP-B (same failure modes), VALVE-1 (Fails to Open, FM0, Mean = 1E-3, EF = 3), and VALVE-2 (same). You now have a 4-component system with multiple failure modes each.

6

Review in Pane F. Click any SSC in Pane C to see its details in the Edit SSC pane (Pane F) on the right. You can edit the name, description, and each failure mode’s parameters directly in this pane. Each failure mode shows its computed probability and distribution parameters.

Loading from a database Instead of adding SSCs manually, you can click Load Database in Pane A to load a .relDB or .fmeaDB file. This populates the Component List tree on the left, from which you can drag components directly into Pane C.

Exercise 2: Writing System Logic

This exercise teaches you to use the “How the System Works” scripting language to define system success criteria and generate fault tree logic.

1

Navigate to Pane E. The “How the System Works” pane shows a text editor with the initial expression SYSTEM WORKS =. Above it, you will see clickable keyword buttons and SSC names that insert tokens at the cursor position.

2

Define a simple redundant system. Type or click to build the expression: SYSTEM WORKS = PUMP-A OR PUMP-B ;. This means the system works if either pump is working (an OR relationship). Press Enter to start a new line, then add: SYSTEM WORKS = VALVE-1 AND VALVE-2 ;. This means both valves must work.

3

Use variables for grouping. Replace the above with a more structured approach. Type: PUMPS= PUMP-A PUMP-B ; (this defines a variable). Then on the next line: SYSTEM WORKS = PUMPS= OR ;. Then: SYSTEM WORKS = VALVE-1 AND VALVE-2 ;. The variable approach makes complex logic easier to read and maintain.

4

Click “Check Script”. The orange Check Script button validates your logic for syntax errors, unmatched parentheses, and unknown identifiers. If it finds SSC names in the logic that don’t exist in your SSC list, it will offer to create them automatically.

5

Try advanced keywords. Experiment with adding dependency logic: PUMPS= RELIES ON POWER ; means the pumps group depends on POWER — if POWER fails, the pumps also fail regardless of their own reliability. Try conditional logic: IF POWER FAILS THEN PUMP-A FAILS ;.

Keyword reference Hover over any keyword button above the editor to see a tooltip with a detailed explanation and usage example. The keywords include SYSTEM WORKS, AND, OR, NOT, RELIES ON, WORKS, FAILS, UNLESS, WHEN, IF/THEN, IS TRUE/FALSE, ARE IN, ARE NOT IN, and comparison operators (=, <, >, <=, >=).

Exercise 3: Analyzing “How the System Fails”

This exercise uses the failure analysis modal to view the derived fault tree, cut sets, and importance measures.

1

Click “How the System Fails” in the toolbar (the dark red button). A full-screen modal opens showing two side-by-side panels. The left panel shows the fault tree derived from your system logic; the right panel shows analysis results.

2

Explore the fault tree. The left panel has two tabs: Text (an indented text representation) and Graphic (a canvas rendering of the tree with AND/OR gates). Toggle between them. The graphic view supports zoom and pan.

3

Review cut sets. In the right panel, click the Cut Sets tab. The minimal cut sets are listed with their probability contributions. Each cut set represents a minimal combination of component failures that causes system failure. The MCUB (Minimal Cut Upper Bound) probability is shown at the top.

4

View importance measures. Click the Importance tab. A sortable table shows each basic event with four importance measures: Risk Increase Ratio (RIR), Risk Reduction Ratio (RRR), Probability of Contribution (PC), and System-Wide Importance Measure (SWIM). Click column headers to sort.

5

Visualize importance. Click the Plot tab. A scatter plot shows two importance measures plotted against each other. Use the axis dropdown menus to change which measures are plotted (e.g., SWIM vs. RIR). Data points are labeled with basic event names. Axis selections are saved for next time.

Linked systems If any SSC uses a “Linked System” probability type (referencing another .KNOW_SYS file), the failure analysis will resolve those linked probabilities automatically. Events with unresolved links show “LINKED” instead of a probability.

Exercise 4: FMEA Editor and Mermaid Diagrams

This exercise demonstrates the Failure Modes and Effects Analysis editor with auto-generated diagrams.

1

Click “FMEA” in the toolbar. The full-screen FMEA modal opens with a two-pane layout: the SSC list on the left and the Mermaid FMEA diagram on the right.

2

Select an SSC. Click PUMP-A in the left pane. The right pane updates to show the FMEA block diagram for this component, rendered using Mermaid.js. Each failure mode is shown as a connected block with its description and probability.

3

Add FMEA details. Click a failure mode in the left pane to expand its editor. Add an Effect (what happens when this failure mode occurs), then fill in Severity, Detection, and Mitigation fields. These fields appear in the FMEA section of reports and in .fmeaDB exports.

4

Add a new failure mode from FMEA. You can add new failure modes directly in the FMEA editor. Click Add Failure Mode for the selected SSC, fill in the failure mode details, and the diagram updates automatically.

5

Review the diagram. The Mermaid diagram is interactive — zoom and pan to explore complex systems. The diagram shows the SSC at the center with failure modes branching out, and effects branching from each failure mode.

Exercise 5: Importing, Exporting, and Reporting

This exercise covers data interchange with other KUREAS modules and report generation.

5A: Importing Data

1

Import boundary conditions from SEQ. Click Import in the toolbar. Select Boundary Conditions from SEQ file. Choose a .KNOW file saved by KUREAS-SEQ. Review the available boundary conditions, check the ones to import, and click Import. Imported boundary conditions appear as tokens in the system logic editor.

2

Import basic events from FTL. Click Import again and select Basic events from FTL file. Choose a .KNOW_FTL file saved by KUREAS-FTL. Review the basic events, select which to import, and confirm. Each imported event becomes an SSC in your system with its probability data preserved.

5B: Exporting Data

3

Export to Reliability Database. Click Export. Select Reliability Database (.relDB). Check the components to include. The exported CSV file contains system name, SSC type, component type, failure mode, description, units, distribution type, and parameters. This file can be loaded back into any KUREAS-SYS project.

4

Export fault tree to KUREAS-FTL. In the Export dialog, select KUREAS-FTL Fault Tree (.KNOW_FTL). This exports the system fault tree (derived from your “How the System Works” logic) as a complete KUREAS-FTL project with basic event probabilities, descriptions, and pre-computed cut sets. Load it directly in KUREAS-FTL for further editing.

5C: Generating Reports

5

Configure the report. Click Report. The report dialog lets you select which sections to include (11 available sections), choose Word (.doc) or Markdown (.md) format, and set a filename. Reorder sections by dragging. Section preferences are saved across sessions.

6

Generate and review. Click Generate. For Word format, a preview opens in the browser where you can review, print, or download. Reports include embedded fault tree graphics, FMEA diagrams, importance measure scatter plots, and all SSC data tables.

7

Save the project. Click Save SYS to save your complete project as a .KNOW_SYS file. This file preserves all system data, SSCs, failure modes, logic, supercomponents, and boundary conditions for future editing or use in KUREAS-REPORT.

Cross-module integration The .KNOW_SYS file can be loaded into KUREAS-REPORT for integrated safety report assembly. The .KNOW_FTL export creates a direct bridge to KUREAS-FTL for detailed fault tree analysis. The .relDB export creates a reusable reliability database for future projects.
